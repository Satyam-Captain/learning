E2E Use Case Rationale and Template:
E2E (end2end) use cases are describing a sequence of steps where each step could be subject to downstream business requirements.
E2E use cases shall feature:
Appropriate attributes associated with its abstraction level, especially: turn-around times, user groups, compliance requirements, frequency they are launched, an owner (for questions), input and output interfaces, role of SPDM, timelines. To understand this, a rationale is explained below:
Turn around times: instead of specifying detailed requirements, business is more driven by turn-around time requirements. It doesn’t help if certain steps are running fast but then data transfers are slowing down the whole process
User group: the number of users is critical information as are user groups. If a capability is also run by externals, then compliance requirements may be confining the use case. Example: ML22 work can today only be executed by externals on Giseh FR due to licensing requirements.
Compliance: If the use case under consideration is also launched on ML22-controlled data, this shall be mentioned. Reason: maybe we need to offer a separate means for ML22 work, so we need to know what the impacted use cases are (could also lead to a decision to offer it on one platform).
Frequency: we need to assess capacity/infrastructure requirements 
Owner: whom to ask in case architects have detailed questions later. 
Timeline/usage profile: it would be good to know when these use cases are launched during aircraft development to identify peak load cases and potential burst strategies (which may depend on compliance drivers!).
Interfaces: we need to identify what happens at the beginning and at the end. Maybe an interface to SPDM is required. Or another E2E use case is launched. 
SPDM interfaces: maybe there are potential exchanges with an SPDM system even while the use cases are running.
Further information: is this use case subject to a potential workflow automation? To be triggered by whom?
Criticality: what happens if this use case is not properly implemented? 
E2E use case description shall contain the sequence of steps and these steps may contain links to detailed business requirements. 
E2E use cases shall provide a detailed description indicating their purpose and role for aircraft design (or other activities).

S1 Modeling & Simulation data storage, accessibility & compliance
Description
This swimlane defines the framework for securing the "Digital Thread" of aircraft design and certification. Its primary purpose is to ensure that every simulation, model, and data point generated during the aircraft's lifecycle is searchable, traceable, and reconstructible for up to 30 years.
For high-level Business Owners, this means mitigating the multi-million Euro risks associated with regulatory non-compliance (e.g., EASA/FAA audits) and intellectual property theft. It transforms data from a "static file on a server" into a "governed asset" that can be instantly retrieved to defend the company's design decisions in the event of an aircraft incident or a patent dispute.

Properties
Data Sovereignty: Enforcement of geolocation and export control (ITAR/ML22) at the metadata level.
System Agnosticism: Decoupling the data (stored on NAS/Cloud) from the management layer (Aras/3DX SPDM).
Archival Integrity: Support for "Cold Storage" retrieval with environment reconstruction (OS/Software versions).
Global Orchestration: High-speed caching across international sites (UK, India, etc.) without losing the "Single Source of Truth."
Automated Governance: Transition from manual file-shuffling to API-driven workflows (Jenkins/Python).

Criticality and Timeline
High. Failure to implement these capabilities results in a direct "Risk of Non-Compliance."
Timeline:
Phase 1 (Immediate): Standardization of Metadata and Metadata-Driven Access Control (MDAC) to replace inflexible legacy access controls.
Phase 2 (Mid-term): Full SPDM Integration to automate the Program Execution Phase (PEP).
Phase 3 (Long-term): Establishing the 15-30 year "Model-Based Certification" environment.

S1UC03 (ALL): Generic Data Orchestration and Caching
Description
Efficiently orchestrates data flows between different geographical locations and compute targets, allowing for local caching while maintaining a single master copy. It is vital that all master data (including results that may be elevated to that status) is captured into the SPDM system so that when files are moved/copied that reference (and its relation to associated metadata) is the one that persists (not the temporary physical location). The location is important because the storage layer is the thing that most adversely affects the overall system performance and end users want this to improve. One way to do that is to make the data closer to the execution/emulation host. This should include potential caching of the user $HOME area to be efficient when running on remote hosts.


Sequence of steps
A user in the UK creates a new file and tags it with metadata (e.g., "aerodynamic mesh"). Upon registration (included with the tagging) the file shall be given an immutable and unique reference identification.
The system automatically copies the file to storage local to the HPC environment for processing. Any use of the file shall use the unique reference identifier for data lineage traceability.
A user in India identifies the need for the file and temporarily caches it locally. Again any use shall incur the unique reference identifier for data lineage traceability.
The master file may be moved into a digital archive or another location to properly protect it depending on the status propagated to it by the SPDM (it could also be deleted if the status becomes “obsolete”).


Users
>6,000 users worldwide. Thousands of concurrent users and API calls must be supported.

Non-Export Control and ML Export Control (ML22).


Interfaces

S1UC04 (ALL): Restricted Data Compliance - Metadata-Driven Access Control (MDAC)
Description
Enforces rigorous data access policies based on multi-axis metadata such as Export Control (EC), Geolocation,Intellectual Property ownership, file status and Military status to ensure regulatory compliance.


Sequence of steps taking UK as an example only
A UK user creates a file tagged as "UK Military". (Note tag list must be standardised and not free text)
The system identifies the tag and restricts the file to the UK environment with write access to the domain team with UK Military accreditation and no world rights.
An accredited UK user accesses the file via a local UK cache.
A user in India attempts access but is blocked by the system based on geolocation and accreditation rules.
The system prevents even an accredited UK user from pulling the file into a cache located in India.
User registers the file in SPDM and turns the status to “published” → system restricts the file as read only to UK users with UK Military accreditation, no world rights and ownership taken by dedicated SA account.

PATMAR is an equivalent case with French National restrictions where the data should be restricted to France.


Note: it is necessary to separate between “Military classification” (which can include weaponisation) and “Military use” (which effectively just identifies the customer type). It is vitally important not to over-classify Airbus Commercial Civil products and to enable maximum civil reuse by derivatives (including for military customers). The digital products must be sustainable on the civil platforms except for weaponisation that must be exported to AD&S.

Users
Global workforce including "Extended Enterprise" (external suppliers/partners).


Owner
Chaima Gouider Eddie Barber SEBASTIEN DROUILHET PEYRE Adrian Chojnacki


Turnaround time
Access rights determined instantly via ABAC (Attribute-based Access Control).


Frequency
Every data access request.


Aircraft Phase
Critical for all phases involving sensitive IP or Military data.


Compliance
Yes; acting on controlled data (ITAR, EC, GDPR).

Interfaces
SPDM for lifecycle rules and ACDS (Airbus Corporate Directory) for identity.


Criticality
Critical; non-compliance carries risks of €20M+ in penalties and loss of reputation.


Pain points
The current system relies on Group-Based Access Control (GBAC), which is too inflexible for complex compliance needs if relied upon on its own (hybrid metadata and GBAC is possible).



S1UC05 (ALL): SPDM control
Description
The SPDM launches workflows and controls the location, context, status and traceability of data throughout the lifecycle. 


Note1: the status changes will influence the Ownership and Groups able to access the files and with which rights (Read/Write/Execute).


Note2: the SPDM could be either 3DX or ARAS and it will be possible to run the entire sequence from that interface for processes where there are no manual interventions required.


Sequence of steps
User selects execution workflow and input files in SPDM (as part of wider process flow) SPDM triggers data to be in the correct location (data written to a cache or existing cache located and manages read only access
User may add files (scripts, models, input data) to the working area (write access)
SPDM or User launches workflow on cached/work area files to create outputs
User uploads results files, any new & dependent input files plus their associated metadata (data lineage, context, etc) to SPDM
SPDM removes the cache/work area once completed or anyway protects the files by taking ownership and making them read only.

Important : Those steps can be triggered by the user on the SPDM or by an application on GISEH doing the same steps on behalf of the user.

S2 - Functional Design Development and Deployment of AFS Applications
Description
This swimlane covers the entire process lifecycle for internal application development, ranging from initial functional design and requirement gathering to engineering, testing, and ultimately, deployment. The primary intent is to standardize and accelerate the delivery of custom-built tools (e.g., pre/post-processing applications, specific analysis automation, data management connectors). It champions the adoption of modern, agile methodologies and DevOps practices to ensure fast, reliable, and compliant deployment of these applications across various computing environments (e.g., Giseh, Cloud, HPC).

Properties
Rapid Development Cycle: Focus on short, iterative cycles aligned with agile and DevOps principles.
Automation-Driven: The core function is to automate or simplify complex engineering workflows, reducing manual effort and potential for errors.
Toolchain Interoperability: Must interface seamlessly with existing COTS CAE software, Simulation Process and Data Management (SPDM) systems, and core data platforms.
Security & Compliance-as-Code: Requires built-in security and compliance checks as an integral part of the development and deployment pipeline.
Diverse Target Environments: Applications must be consistently deployable to multiple operating systems and computing environments (Linux, Windows, VDI, HPC, and Cloud).
Prioritisation of batch - it shall be possible to run anything in batch and the means to use batch processing shall be simplified as far as possible (currently it is much easier to run on GISEH interactively and this must be turned around). 
End user developments shall be massively increased due to python and digital transformation → need to consider how to accommodate this and control quality, maintainability, etc!

Criticality and Timeline
High. This capability is foundational to realizing the benefits of process automation and digital continuity across all engineering swimlanes. The adoption of a structured DevOps pipeline is essential for compliance and maintaining the velocity for new capabilities required for eAction milestones. Initial deployment of the Continuous Integration/Continuous Delivery (CI/CD) framework is required within the next milestone, with continuous application releases targeted immediately thereafter.


S2UC01 (1YC, 1YV): Development of Giseh compatible solution  
Description
Engineering developers require a seamless workflow to develop, test, and deploy software solutions that are guaranteed to function across multiple Airbus target environments (e.g., GISEH, CBCC, SysDops/ePaaS, and Local Workstations). The business goal is to eliminate "environment friction"—where code works on a developer's machine but fails on target infrastructure due to mismatched dependencies, outdated compilers, or incompatible container runtimes. The process must support both custom-built code and the integration of containerized commercial (COTS) engineering tools.
Sequence of Steps
Environment Initialization: Developers initialize a project using Airbus-approved standard dependencies (Conda, Maven, etc.) that are mirrored across all target platforms.
Local Development: Developers perform core coding and local execution on their primary workstation to leverage high-performance IDE features and local debugging tools.
Cross-Platform Validation: Before finalizing development, the developer triggers automated tests that simulate or mirror the specific constraints of the target environments (e.g., GISEH’s RHEL version or SysDops’ security posture).
Integrated Data Access: During development and testing, the solution must securely access engineering datasets (e.g., on Network Attached Storage) without requiring manual data duplication or complex tunneling.
Automated Deployment: Once validated, the solution (including any necessary commercial tool containers) is pushed to a central registry.
Target Execution: The solution is deployed to the production environment (Giseh, Cloud, or On-Prem) and runs "first-time-right" because the development environment was perfectly aligned with the execution environment.

Modern Tooling: Integration with modern IDEs (VS Code, JetBrains CLion/PyCharm, Zed) and AI coding agents (Gemini CLI, Claude Code, Codex).

Compilation: Developer triggers parallel builds (make -j or ninja), utilizing the dedicated 128GB+ RAM and multi-core CPUs to compile large C++/Fortran codebases without Out-Of-Memory (OOM) errors.
Testing & Debugging: Developer runs test binaries against the newly generated meshes and connects parallel debuggers (like DDT, via its remote client-server mode) directly to the remote process for fluid, low-latency debugging. S4 - Workflow design and execution
Description
This swimlane encompasses all workflow-based business activities which are used to automate tasks that would ordinarily be performed by an engineer individually or in a team. It provides the infrastructure and software patterns required to transform complex manual design tasks (like airframe sizing or acoustic prediction) into reproducible, automated pipelines. This covers the entire lifecycle of a workflow from design and modification through to running (on-demand or automated) and retirement as well as the creation of bespoke and reusable workflow components. The swimlane is also concerned with how the data produced by the workflow is interfaced with the approved SPDM system(s).

Properties
Automated: The system runs tests or tasks without requiring manual intervention, once configured.
Headless Execution: Transitioning "heavy" GUI-based tools (HyperWorks, Isight, Eclipse) into "No-GUI" batch processes that run in the background.

S6 - Modeling & Simulation using Artificial Intelligence, Machine Learning and Surrogate Modelling
Description
This swimlane defines the framework for hybrid modeling, where physical simulations are augmented or replaced by Machine Learning (ML) and Surrogate Models. The intent is to leverage historical flight data (Skywise), test bench results, and high-fidelity simulation data to create "lightweight" mathematical models (Surrogates).
For high-level Business Owners, this swimlane represents Speed and Competitive Agility. Traditional simulations can take days; a trained surrogate model can provide an answer in milliseconds.

Properties
Hardware-Accelerated Compute: Requirements for GPU integration (NVIDIA/CUDA) for model training and high-speed inference (Note: modern high-performance GPU integrations NVidia GB200+, Vera Rubin etc., ship ARM64-based CPU backbones and therefore are not compliant with the current x86 codebase in use at Airbus).
Hybrid Data Connectivity: Bidirectional bridges between On-Premise (GISEH), Cloud (AWS/Skywise), and specialized repositories (Artifactory).
Training & Inference: A platform must account for the generally different workload requirements between model training (parallel high-end GPU, HBM memory & storage bound checkpointing) and model inference (batch & online inference). The target platform may differ between the fundamental use cases of training and inference.
MLOps Lifecycle: Standardized workflows for model versioning, training, publishing (of datasets and models), and retirement (similar to software CI/CD but for AI).
Location-Aware Scheduling: "Data-proximate" computing—bringing the math to the data to avoid moving massive, sensitive datasets across borders.
IP Protection: Rigorous isolation of of some data (e.g. powerplant manufacturer data and military-grade export control) within the ML training environment.

Criticality & Timeline
High. This is no longer a "nice-to-have" research project; it is a core requirement for the next generation of aircraft. Without these capabilities, Airbus cannot meet the architectural phase deadlines for new programs next year.
